# SalesPilot — Multi-Agent Sales Data Assistant

## Overview

Sales reps ask business questions in natural language. A LangGraph orchestrator
classifies intent and routes to specialized sub-agents that query structured data
(Postgres) and contract documents (ChromaDB). A synthesis node returns a grounded
answer in which **every factual claim is annotated with its source** (table name
or document name). If the data isn't available, the system says so — it never
fabricates.

## Tech stack

- Python 3.11+, LangGraph, langchain-anthropic (Claude Sonnet)
- PostgreSQL (psycopg2), ChromaDB (local persistent client)
- Langfuse Cloud for tracing
- FastAPI + minimal chat UI for the demo surface

## Repo layout

```
salespilot/
  CLAUDE.md
  seed_data.py          # seeds 5 tables + writes contract_docs/*.md
  contract_docs/        # ~30 contract markdown files (generated by seed script)
  graph/
    state.py            # AgentState TypedDict
    orchestrator.py     # intent classification node
    sql_agent.py        # NL -> SQL -> read-only execution
    rag_agent.py        # ChromaDB retrieval
    synthesis.py        # grounded answer assembly
    build.py            # graph wiring (nodes + conditional edges)
  ingest_contracts.py   # chunk contract_docs/ into ChromaDB
  main.py               # CLI entry: python main.py "question here"
  tests/
    test_acceptance.py  # end-to-end acceptance questions
```

## Data layer

`seed_data.py` creates and populates: `accounts` (30), `contacts`, `products` (50),
`orders` + `order_items` (~192 orders across 12 months), `contracts` (per-account
discount_rate, renewal_date). Run with `--verify` to sanity-check the dataset
against the acceptance queries. Contract docs are written for every account (~30 files
in `contract_docs/`); only the 6 named accounts have meaningful discount/pricing terms.

Dataset invariants the acceptance tests depend on:
- "Acme Corp" exists, contract discount 0.120, doc `acme_corp_msa.md`
- Exactly 5 dormant accounts (no order in 90+ days)
- Product `PX-1000` list price 1250.00; Acme's MSA fixes it at 1100.00
- `order_items.unit_price` stores the discounted price actually charged

## Architecture

### AgentState

```python
class AgentState(TypedDict):
    question: str
    required_agents: list[str]      # subset of ["sql", "rag"] — a list, because
                                    # cross-source questions need both agents
    sql_result: dict | None         # {"query": str, "rows": list, "tables": list[str]}
    rag_result: dict | None         # {"chunks": list, "sources": list[str]}
    final_answer: str
```

### Graph shape

orchestrator -> conditional edges fan out to sql_agent and/or rag_agent
(parallel branches when both are required) -> branches converge on synthesis
-> END. The orchestrator outputs `required_agents`; the conditional edge
function returns next nodes based on it.

### SQL safety — two layers, both mandatory

1. Application layer: validate generated SQL with sqlparse — single statement
   only, statement type must be SELECT; reject INSERT/UPDATE/DELETE/DROP/
   ALTER/GRANT/COPY and statement chaining.
2. Database layer: execute through the `sp_readonly` role (SELECT-only grants,
   created by seed_data.py). The agent's connection string uses sp_readonly,
   never the owner user.

### Grounding rules

- Every numeric/factual claim in final_answer cites its source inline:
  `(source: orders table)` or `(source: acme_corp_msa.md)`.
- The SQL agent returns the list of tables it touched; the RAG agent returns
  doc filenames from chunk metadata. Synthesis may only cite these.
- If neither agent returned relevant data, answer "I could not find this in
  the available data" — never fabricate or estimate.

### RAG ingestion

Chunk `contract_docs/*.md` by markdown section (`##` headers) into a persistent
ChromaDB collection. Metadata per chunk: {filename, account_name, section_title}.
Retrieval: top-4 chunks.

## Implementation plan

1. **State + graph skeleton** — `state.py` + `build.py` with stub nodes;
   conditional edges wired. Verify: `python main.py "test"` runs end-to-end.
2. **Orchestrator node** — structured `required_agents` output via tool use.
   Verify: acceptance questions classify correctly (Q4 -> ["sql", "rag"]).
3. **SQL agent** — schema injection, SQL generation, two-layer read-only
   validation, execution. Verify: Q1/Q3 return correct rows; write-statement
   prompts are rejected.
4. **RAG agent** — `ingest_contracts.py` + retrieval node. Verify: Q2 retrieves
   the Acme discount clause with correct filename metadata.
5. **Synthesis node** — grounded answer assembly per the grounding rules.
   Verify: all acceptance questions pass; unanswerable questions get an
   honest "not found".
6. **Langfuse tracing** — CallbackHandler wired into graph invocation.
   Verify: full multi-agent trace visible with all node spans.

UI and deployment follow after steps 1-6 pass.

## Acceptance tests

1. "Which accounts haven't ordered in 90 days?" → exactly the 5 dormant accounts
2. "What's the contract discount for Acme Corp?" → 12% with source cited
3. "Top 5 products by revenue this quarter?" → matches `seed_data.py --verify`
4. "Compare Acme's contract price vs catalog price for product PX-1000" →
   $1,250 (products table) vs $1,100 (acme_corp_msa.md), both sources cited

## Engineering conventions

- Implement and verify one plan step at a time; each step's verification must
  pass before the next begins.
- Keep modules small and single-purpose. No premature abstraction.
- Never connect the SQL agent with a writable database role.
